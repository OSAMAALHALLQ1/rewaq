# Context: Rewaq SaaS Multi-Department Access & Messaging

This document defines the ubiquitous language, domain terms, and core design decisions for the multi-department permission control and internal messaging system in Rewaq SaaS.

## Glossary

### Department API Key (رمز وصول القسم)
A unique 10-character alphanumeric token (e.g., `RWQ_A7B8C9`) generated by the Organization Owner (Restaurant Manager). It authorizes a specific department terminal to access designated APIs and pages without requiring email/password credentials.

### Dedicated Access Link (رابط الوصول المخصص)
A direct URL formatted with the Department API Key as a query parameter (e.g., `/d/gate?key=RWQ_A7B8C9`). Opening this link on a terminal automatically authenticates the device, sets up the session cookie or local storage, and grants instant access to the specific department's workspace.

### Department Terminal (جهاز القسم)
A fixed device or display (e.g., Kitchen Display System (KDS), POS Cashier Tablet, or Inventory Terminal) operating within a specific branch and assigned a single restricted role.

### Internal Message (الرسائل الداخلية)
A real-time notification or message sent within a restaurant between departments or individuals (e.g., Cashier to Kitchen, or Kitchen to Warehouse) powered by Supabase Realtime for instant communication.

## Architectural Decisions

### ADR 0001: 10-Symbol Alphanumeric Department Keys
We use a 10-character alphanumeric key generated by the organization owner for authenticating department devices via specific gateway URLs. This eliminates the need for username/password management on shared devices while maintaining secure, auditable, and easily revocable tokens.

### ADR 0002: Permanent Key Validity Until Manual Revocation
Department API Keys remain active indefinitely to ensure absolute uptime and stability for physical restaurant hardware (KDS, POS tablets). However, the Organization Owner maintains full control to manually revoke, deactivate, or regenerate any key instantly from the admin settings dashboard.

### ADR 0003: Full-Screen, Role-Specific Terminal UI
Department terminals accessing via a dedicated link are presented with a simplified, full-screen UI optimized for tablet/workstation views. The global SaaS sidebar, search, and navigation controls are hidden. Instead, the layout renders a focused, single-purpose application interface matching the department's role (e.g., KDS view for Kitchen, POS sell view for Cashier) to prevent distractions and interface errors.

### ADR 0004: Hierarchical Communication with Manager Oversight
Internal messaging is structured into:
1. **Department Channels**: Focused communications directed from one department to another (e.g., POS-to-Kitchen, Kitchen-to-Warehouse) where only members/terminals of the sender and recipient roles can read/write.
2. **General Restaurant Group**: A branch-wide or organization-wide channel accessible by all staff for general announcements and collaborative chat.
3. **Manager Administrative Oversight**: The Organization Owner (Manager) has universal access to view all chat history across both department-specific channels and the general group, ensuring complete operational oversight and quality control.

### ADR 0005: Dynamic Custom Permission Mapping per Device Key
Rather than relying on rigid, pre-defined server roles alone, the Organization Owner (Manager) can selectively assign granular module permissions (e.g., Inventory, Recipes, Waste Logging, POS, Purchasing) to a Department Key upon creation. The system dynamically restricts client-side navigation elements (hiding/showing links) and enforces server-side route guards matching this list of allowed modules, allowing maximum operational flexibility.
