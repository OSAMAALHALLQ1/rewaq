هذه الخطة لا تفترض أن التقرير يعكس آخر نسخة من الكود. يجب إثبات حالة كل بند من الكود الحالي والمهاجرات والاختبارات قبل تنفيذ أي تغيير. لا تطبق مهاجرات على الإنتاج تلقائيًا، ولا تحذف سجلات مالية أو مخزنية أو تدقيقية.

- كل قيد مرحل متوازن ويستخدم تاريخ المستند ويرفض الفترة المقفلة.
- التقارير المالية تعتمد القيود ذات الحالة `posted` فقط وتطابق دفتر الأستاذ.
- كل إصلاح يملك اختبار رجوع، ويمر عبر typecheck وlint والاختبارات والبناء.

## تحديث 11 يوليو 2026

تم إغلاق خلل محاسبي حرج في دفعات الموردين على مستوى الكود:

- أضيفت migration رقم `044_atomic_supplier_payments.sql` بعد إعادة ترقيمها لتفادي تعارضها مع ترحيلات GitHub الجديدة `040` إلى `043`.
- أضيفت دالة `record_supplier_payment_atomic` لتسجيل الدفعة، قيد اليومية، تحديث رصيد الفاتورة، وسجل التدقيق في عملية قاعدة بيانات واحدة.
- أضيف `idempotency_key` لجدول `supplier_payments` مع فهرس فريد لكل مؤسسة.
- صار قيد دفع المورد يرتبط بمستند الدفعة نفسها بدل الفاتورة، حتى تدعم الفاتورة الواحدة عدة دفعات جزئية دون إعادة استخدام نفس القيد.
- تم ربط إجراء دفع المورد في التطبيق بالـ RPC الجديد.
- تم تنظيف تواريخ قيود POS التجريبية لاستخدام `todayLocal()` بدل تاريخ UTC.

لم يتم تطبيق migration على Supabase الإنتاجي. قبل التطبيق على أي عميل، نفذها أولاً على مشروع Supabase تجريبي ثم تحقق بهذه الاستعلامات:

```sql
-- لا توجد دفعتان بنفس مفتاح التكرار داخل نفس المؤسسة
select organization_id, idempotency_key, count(*)
from supplier_payments
where idempotency_key is not null
group by organization_id, idempotency_key
having count(*) > 1;

-- كل دفعة مورد لها قيدها الخاص
select sp.id as payment_id, sp.invoice_id, sp.amount, je.id as journal_entry_id
from supplier_payments sp
left join journal_entries je
  on je.organization_id = sp.organization_id
 and je.source_doc_type = 'supplier_payment'
 and je.source_doc_id = sp.id
where sp.journal_entry_id is null
   or je.id is null;

-- أرصدة فواتير الموردين تساوي مجموع الدفعات المسجلة
select i.id, i.total, i.paid_amount, i.balance_due, coalesce(sum(sp.amount), 0) as payments_total
from invoices i
left join supplier_payments sp
  on sp.organization_id = i.organization_id
 and sp.invoice_id = i.id
group by i.id, i.total, i.paid_amount, i.balance_due
having abs(coalesce(i.paid_amount, 0) - coalesce(sum(sp.amount), 0)) > 0.01
    or abs(coalesce(i.balance_due, 0) - greatest(0, coalesce(i.total, 0) - coalesce(sum(sp.amount), 0))) > 0.01;
```

حالة P0 بعد هذا التحديث:

- `P0-05` تحسنت بوضوح لمسار الدفعات، وتحتاج تحقق حي على Supabase تجريبي قبل اعتبارها مغلقة بالكامل.
- `P0-07` ما زالت مفتوحة جزئياً لأن استلام طلبات الشراء وفواتير الموردين لا تزال تحتاج RPC واحد يغطي المخزون، المستند، GRNI، وتحديث الحالة دفعة واحدة.
- `P0-08` ما زالت تحتاج اختبار تكامل حي للاستلام، رغم وجود مفاتيح idempotency.
