# سجل تقدم رواق نحو الجاهزية للبيع

آخر تحديث: 2026-07-17

## الهدف التنفيذي

تجهيز رواق كنظام عربي متكامل لإدارة مطعم واحد أولاً، مع مسار توسع آمن للفروع، وربط دورة العمل من طلب الجرسون إلى محطات التحضير والتجميع والتسليم والفاتورة والمخزون والقيد المحاسبي. يجب أن تكون الوحدات والصلاحيات والباقات مطبقة فعلياً في الخادم وقاعدة البيانات، لا مجرد عناصر واجهة.

## ميزة رواق الفارقة

الميزة ليست كثرة الشاشات، بل «برج تشغيل رواق»:

1. كل عملية بيع مرتبطة بالطلب والطاولة والجرسون ومحطات التحضير.
2. كل صنف يستهلك وصفته من المخزون مع سجل حركة قابل للتدقيق.
3. النقص يولد توصية شراء قابلة للتحويل إلى طلب شراء.
4. الاستلام وفاتورة المورد والدفع تنتقل إلى دفتر الأستاذ دون إدخال مكرر.
5. صاحب المطعم يرى الاستثناءات القابلة للتنفيذ: طلب متأخر، صنف جاهز لم يسلم، نقص مخزون، هدر مرتفع، فرق صندوق، فاتورة بلا قيد، أو قيد غير متوازن.
6. التحليلات تعتمد على بيانات فعلية؛ عند غياب البيانات تظهر حالة واضحة بدلاً من أرقام مصطنعة.

## دمج فكرة تكة

سيتم نقل الفكرة وليس البنية القديمة:

جرسون -> طلب موحد -> تذاكر حسب المحطة -> KDS -> Expo -> تسليم -> كاشير -> فاتورة عميل -> مخزون -> قيد

قرارات الدمج:

- إعادة استخدام جداول الطلبات وبنود الطلب وتذاكر المطبخ والطاولات الموجودة في رواق.
- توجيه كل بند حسب محطة التحضير، لا إنشاء نسخة منفصلة من الطلب لكل قسم.
- منع حذف تذاكر الأقسام بعد الدفع؛ تغلق بحالة نهائية ويحفظ أثرها.
- استخدام مفتاح idempotency لكل إرسال ودفع لمنع التكرار عند إعادة المحاولة أو العمل Offline.
- المحاسب لا يعيد إدخال الفاتورة؛ يرى مستند البيع والقيد الناتج وحالة المطابقة.

## الباقات الشهرية

| الباقة | السعر | الوحدات الأساسية |
|---|---:|---|
| رواق تشغيل | 150 دولار | POS، ورديات وصندوق، طاولات، جرسون، KDS/Expo، عملاء، فواتير مبيعات، تقارير تشغيل أساسية |
| رواق إدارة | 250 دولار | كل ما سبق + مخزون، وصفات، تكلفة طبق، هدر، موردون، مشتريات، تحويلات، تقارير ربحية وموافقات |
| رواق متكامل | 350 دولار | كل ما سبق + المحاسبة الكاملة، مراكز التكلفة، الإقفال، التسويق والنشر، الأتمتة، التحليلات المتقدمة وكل الوحدات |

مبادئ التطبيق:

- القفل يطبق في Server Actions وAPI وقرارات النطاق، وليس في القائمة فقط.
- إخفاء الوحدة غير كاف؛ أي طلب مباشر لمسار مقفل يجب أن يرفض.
- تخفيض الباقة لا يحذف بيانات قديمة. تصبح البيانات للقراءة أو التصدير وفق سياسة الوحدة.
- الباقة العليا تفتح جميع الوحدات الحالية.
- حدود الفروع والمستخدمين والأجهزة تحفظ كبيانات خطة قابلة للتعديل، لا أرقام موزعة داخل الواجهة.

## الاستضافة الأولية

- Vercel Hobby للاستخدام الشخصي غير التجاري فقط، لذلك يصلح للعرض والتطوير ولا يصلح لتشغيل عميل مدفوع.
- أول عميل تجاري يحتاج Vercel Pro أو مضيفاً يسمح بالاستخدام التجاري.
- Supabase Free يمنح مشروعين مجانيين، مع 500MB لقاعدة البيانات لكل مشروع و1GB تخزين وقيود استخدام أخرى.
- رواق سيحتوي مراقبة نمو البيانات وتنبيه ترقية قبل بلوغ الحدود.

المصادر:

- <https://vercel.com/docs/plans/hobby>
- <https://supabase.com/docs/guides/platform/billing-on-supabase>

## فريق التنفيذ الحالي

1. هندسة POS/KDS: مسار تكة ومحطات التحضير ودورة الحالات.
2. الباقات والأمان: الاستحقاقات، الصلاحيات، RLS وقفل الخادم.
3. المحاسبة: الفاتورة والقيد والذرية ومنع التكرار.
4. واجهة RTL وQA: تجربة القفل والترقية والهاتف واللوحي.

## بوابة البيع

لا يوصف النظام بأنه جاهز للبيع قبل تحقق جميع الآتي:

- نجاح TypeScript وLint والاختبارات وBuild وPlaywright.
- تطبيق المهاجرات الجديدة على Supabase تجريبي فقط وتشغيل فحوص RLS وRPC.
- نجاح دورة بيع كاملة حتى المخزون والقيد المحاسبي.
- نجاح دورة مرتجع وعكس محاسبي دون حذف.
- نجاح عزل مؤسستين ومنع الوصول المباشر لوحدة مقفلة.
- عدم وجود زر أو فلتر أو شاشة تمهيدية بلا وظيفة.
- توثيق النسخ الاحتياطي والاستعادة وحدود الاستضافة ومسار الترقية.
