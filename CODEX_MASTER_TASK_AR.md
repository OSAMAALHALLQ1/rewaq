# المهمة الرئيسية لـ Codex: تجهيز رواق للعمل المنضبط بالوكلاء

أنت تعمل داخل مستودع **رواق**، وهو نظام متعدد المؤسسات لإدارة المطاعم والكافيهات، ويشمل POS والمخزون والوصفات والإنتاج والمشتريات والمحاسبة والصلاحيات والتسويق.

## الهدف

جهّز المستودع لاستخدام Codex وClaude Code بطريقة آمنة ومنظمة، اعتمادًا على الملفات الموجودة في هذه الحزمة. لا تبدأ بإعادة بناء المنتج، ولا تنفذ تغييرات تجارية كبيرة. المطلوب أولًا هو **تهيئة بيئة الوكلاء، التعليمات، الاختبارات، MCP وCI** مع احترام البنية الحالية.

## قواعد إلزامية

1. افحص المستودع الحالي قبل أي تعديل.
2. لا تستبدل ملفًا موجودًا دون مقارنته ودمج المحتوى يدويًا.
3. أنشئ نسخة احتياطية نصية لأي ملف إعداد ستعدله.
4. لا تعرض أو تنسخ أو تسجل أي سر.
5. لا تستخدم `SUPABASE_SERVICE_ROLE_KEY` في الواجهة أو الاختبارات العامة.
6. لا تتصل بقاعدة الإنتاج بصلاحية كتابة.
7. لا تطبق Migration قبل عرض SQL وخطة rollback.
8. لا تحذف مستندات مالية مرحلة؛ التصحيح يكون بالعكس أو إشعار دائن/مدين.
9. كل استعلام أعمال يجب أن يكون مقيّدًا بـ`organization_id`، وبالفرع/المستودع عند الحاجة.
10. إخفاء زر في الواجهة ليس صلاحية؛ التحقق يجب أن يكون على الخادم وRLS.
11. لا تضف Dependency إنتاجية دون سبب واضح وموافقة.
12. لا تعمل مباشرة على `main`؛ استخدم Branch أو Worktree.
13. لا تعلن النجاح قبل تشغيل الاختبارات الفعلية.

## المرحلة 1: اكتشاف البيئة

نفّذ ثم لخّص النتائج:

- بنية المشروع وStack الفعلي.
- مدير الحزم وملف القفل.
- أوامر `package.json` الحالية.
- إعداد Next.js وTypeScript وESLint والاختبارات.
- وجود Playwright/Vitest/Jest.
- بنية Supabase والمهاجرات وRLS.
- ملفات `AGENTS.md` و`CLAUDE.md` و`.mcp.json` الحالية.
- GitHub Actions الحالية.
- أي Skills أو Subagents موجودة.

لا تثبت شيئًا في هذه المرحلة.

## المرحلة 2: دمج تعليمات المشروع

ادمج محتوى:

- `AGENTS.md`
- `CLAUDE.md`

مع الملفات الحالية إن وجدت. حافظ على قواعد المشروع الموجودة ولا تكررها. أضف أو حدّث Skills داخل:

```text
.agents/skills/
```

وتأكد أن كل Skill تحتوي `SKILL.md` صالحًا مع `name` و`description` فريدين.

أضف Subagents الخاصة بـClaude داخل:

```text
.claude/agents/
```

لا تجعل وكلاء المراجعة يملكون صلاحية كتابة إن لم يحتاجوا إليها.

## المرحلة 3: Playwright

إذا لم يكن Playwright موجودًا:

1. استخدم مدير الحزم الموجود في المشروع.
2. أضفه كـdev dependency فقط.
3. ثبّت Chromium على الأقل.
4. أنشئ إعدادًا لا يكسر المشروع الحالي.
5. أضف Smoke Test لتسجيل الدخول أو الصفحة العامة دون تضمين بيانات اعتماد حقيقية.

إن كان موجودًا، لا تعِد تهيئته؛ حسّن الإعداد الحالي فقط.

المسارات التي يجب التخطيط لاختبارها لاحقًا:

- تسجيل الدخول وعزل المؤسسة.
- فتح وردية وإنشاء طلب ودفعه.
- طاولة ← مطبخ ← جاهز ← دفع.
- أمر شراء ← استلام ← فاتورة مورد ← دفع جزئي.
- جرد ← اعتماد فرق ← حركة مخزون وقيد.
- تالف ← موافقة ← خصم مخزون وقيد.
- منع مستخدم فرع A من بيانات فرع B.
- منع التكرار عند إعادة إرسال الطلب.

## المرحلة 4: MCP بوضع آمن

### Supabase

اقترح إعدادًا مقيّدًا بمشروع رواق:

```text
https://mcp.supabase.com/mcp?project_ref=<PROJECT_REF>&read_only=true
```

لا تضع `<PROJECT_REF>` الحقيقي أو Access Token في ملفات Git. استخدم Environment Variables أو OAuth. لا تفعّل الكتابة حتى يراجع المستخدم الإعداد.

### GitHub

استخدم GitHub MCP الرسمي. ابدأ بوضع Read-only وأقل Toolsets لازمة:

```text
context,repos,issues,pull_requests,actions,code_security
```

لا تطلب صلاحيات إدارة Secrets أو حذف فروع أو دمج تلقائي.

### Context7

أضف Context7 لتوثيق المكتبات الحديث، أو استخدم:

```text
codex mcp add context7 -- npx -y @upstash/context7-mcp
```

لا تثبت نسخًا مجهولة أو Packages بأسماء مشابهة.

### Playwright MCP

بعد تثبيت اختبارات Playwright، يمكن إضافة MCP للاستكشاف:

```text
codex mcp add playwright npx "@playwright/mcp@latest"
```

ولـClaude:

```text
claude mcp add playwright npx @playwright/mcp@latest
```

اعرض الأوامر للمستخدم قبل تنفيذ أي OAuth أو إعداد يتطلب حسابًا.

## المرحلة 5: Hooks وCI

راجع `.claude/settings.example.json` ولا تنسخه حرفيًا إذا تعارض مع المشروع. أنشئ إعداد Hooks يقوم فقط بأوامر آمنة مثل:

- typecheck إن كان موجودًا.
- lint إن كان موجودًا.
- اختبار ملفات معدلة عند الإمكان.
- فحص الأسرار قبل الإنهاء.

ادمج Workflow `rewaq-quality.yml` مع GitHub Actions الحالية. يجب أن يشغّل ما هو موجود فعليًا فقط، ولا يخترع أوامر غير معرفة.

الحد الأدنى المطلوب لكل Pull Request:

- تثبيت dependencies بطريقة reproducible.
- typecheck.
- lint.
- tests.
- build.
- Playwright smoke عند توفر البيئة.
- CodeQL منفصل إن كان GitHub Advanced Security أو الإعداد المناسب متاحًا.

## المرحلة 6: التحقق النهائي

شغّل:

- الأمر الفعلي للـtypecheck.
- lint.
- unit/integration tests.
- build.
- Playwright smoke.

ثم قدم تقريرًا منظمًا:

1. الملفات التي أُضيفت.
2. الملفات التي عُدّلت.
3. الأدوات المثبتة وإصداراتها.
4. MCP الذي تم إعداده والذي يحتاج OAuth.
5. الاختبارات التي نجحت.
6. الاختبارات التي فشلت مع السبب.
7. المخاطر أو التعارضات المكتشفة.
8. ما لم يُنفذ ولماذا.
9. الخطوة التالية المقترحة.

## ممنوعات

- ممنوع تعديل بيانات الإنتاج.
- ممنوع تشغيل أوامر حذف قاعدة البيانات.
- ممنوع استخدام `--force` أو تجاوز فشل الاختبارات بلا تفسير.
- ممنوع وضع مفاتيح في `.mcp.json` أو `AGENTS.md` أو Logs.
- ممنوع تحويل كل خطأ خادم إلى Offline Queue.
- ممنوع اعتماد دور المستخدم من الواجهة فقط.
- ممنوع إنشاء نظام محاسبي أو مخزون موازٍ للنظام الحالي.
